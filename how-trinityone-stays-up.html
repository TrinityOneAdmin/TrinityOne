TrinityOne Back
Architecture · semi-technical

How TrinityOne stays up.

An honest walkthrough of how a church's life on TrinityOne keeps working when things go wrong — a host going down, a domain blocked, a state filtering an address. Written so somebody who's never run a server can follow the main flow, and somebody who's done deep protocol work can still find the details. Technical asides are clearly marked as For the pro and can be skipped on a first read.

One-paragraph TL;DR. TrinityOne is built in layers — chat & data on Nostr relays, modules (Bibles, dictionaries) downloaded from multiple mirrors keyed by content hash, and the app itself installable from any TrinityOne church's relay. Each layer has its own redundancy. We've also designed (built but not yet shipped) a Blossom layer that addresses module bytes by their SHA-256 hash and rides on signed Nostr events, so module distribution survives even when the canonical host is blocked. None of this requires the user to do anything — it's about what's still possible when a thing goes dark.

The problem this design solves

Most church software is one company's web app on one company's servers. When that company has a bad day (the website's down), a bad year (they get bought, prices go up), or a bad incident (an authority orders them to remove a customer), every church on the platform feels it. TrinityOne is built so a church actually owns what it builds on TrinityOne — its chats, its members, its giving records — and that ownership only matters if the data and the app itself stay reachable.

So a sensible question is: what does it take to take TrinityOne down? The answer should be a longer list than "press one button." This document explains why.

The three layers, in plain language

TrinityOne has three distinct things that need to stay reachable. Each has its own redundancy mechanism — they don't all rise and fall together.

┌─────────────────────────────────────────────────────────────────┐ │ Layer 3 — Getting the app in the first place │ │ ↳ Marketing page, APK download, "Add to Home Screen" URL │ │ ↳ Every church's relay also serves the app │ │ ↳ One person with the APK can sideload to many │ ├─────────────────────────────────────────────────────────────────┤ │ Layer 2 — Modules: Bibles, dictionaries, catalogue │ │ ↳ Files downloaded once, cached forever on the device │ │ ↳ Hash-verified so a wrong-bytes mirror is rejected │ │ ↳ "Blossom" design: any host with the file serves it │ ├─────────────────────────────────────────────────────────────────┤ │ Layer 1 — Chat & data — the day-to-day │ │ ↳ Nostr relays your church and its network run │ │ ↳ Multiple relays; clients try all of them │ │ ↳ Your phone is itself a full copy of your church's data │ └─────────────────────────────────────────────────────────────────┘

Layer 1 was always relay-based — it's what makes the rest possible. Layer 2 was originally one host serving files over HTTP; we've redesigned it to be hash-addressed and multi-host. Layer 3 is human-and-mesh, with documentation we've shipped showing people how to install from any church's relay.

Layer 1: Relays — the foundation

What a relay actually is

A relay is a small server program — small enough that you could run one on a Raspberry Pi sitting under the desk in a church office. It does two things: it stores signed messages that someone sends it, and it sends them on to anyone subscribed. That's it. There's no user database, no login, no password, no payment system. It's plumbing.

Every chat message, every group definition, every fund record, every "I'm a steward of this church" doc — they're all small files (called events) signed by the key that wrote them. The relay just stores and forwards. The signatures prove what came from whom.

For the pro NIP-01 over WebSocket; events keyed by kind + the addressable d-tag (for replaceable kind-30078 docs) or by id (kind-1, kind-4, etc.). The TrinityOne pilot relay is scripts/gateway.mjs — a custom Node WS implementation that does NIP-01 plus an embedded write-policy enforcer (multi-church, group leaders, safeguarding DM gate, blocklist). It's replaceable for production with Khatru, strfry, or nostr-rs-relay. The NAS pilot relay does in fact run nostr-rs-relay (in Docker) instead of our gateway — same Nostr surface, simpler write policy.

Why multiple relays matter

If your church's data only lived on one relay, that relay is a single point of failure. So TrinityOne clients (the member app, the steward console) carry a list of canonical relays they all talk to. When a member sends a message, it goes to every relay on their list. When they read messages, every relay's events get merged. Lose one, the others carry it.

There's a further wrinkle — your own phone is also a complete copy. Every event you've seen, you have. If every TrinityOne relay went dark tomorrow, your church's recent life is still on your steward's device, ready to re-seed a fresh relay the moment one comes back.

For the pro CANONICAL_RELAYS is defined in src/fellowship.src.js (member side) and src/steward.src.js (steward side); both bundled into vendor/*.js. SimplePool from nostr-tools handles the multi-relay subscribe/publish. The "phone as a copy" claim is literal: the member's IndexedDB cache holds every event their subs have ever delivered, and the steward console can re-publish them to any pool. We don't yet expose a one-click re-seed UI; that's the parked Keykeeper/relay-resilience work (see reference/SPINE.md).

Layer 2: Modules — Bibles, dictionaries, the catalogue

Modules are files: a Bible translation in USFM (an open standard), Strong's lexicon as JSON, a MySword-format dictionary as SQLite. They're downloaded once and cached on the device — re-opening a verse you've read before is offline-friendly.

What was wrong with the original design

Originally, the member app fetched everything from one host: the catalogue (the list of available modules), each module file, and the marketing page. Block that one host, and members couldn't even see what was available, let alone download it. Even though the actual content — Strong's, the KJV, BSB — is public domain and unblockable in principle, the delivery path was a single point.

Phase 1 — multi-mirror with content hashes

The first design change: every catalogue entry now carries its SHA-256 hash plus a list of mirror URLs. The client tries each mirror in turn; whichever returns the bytes is verified against the hash. A mirror serving wrong bytes is rejected, and the next is tried.

{
  "id": "strongs",
  "name": "Strong's Greek & Hebrew Dictionary",
  "url": "modules/strongs-dict.json",
  "sha256": "8a2a130d8e0f2c0ec22bd1891c186cd9eed9477152e57f582ed0dbfb6f3769c1",
  "bytes": 4012877,
  "mirrors": [
    "https://trinityone.tailbeaac0.ts.net/modules/strongs-dict.json",
    "https://trinityone-master-01.tailbeaac0.ts.net/modules/strongs-dict.json"
  ]
}

Why this is a real win: the church relays you read about in Layer 1 already serve files (the relay-app installer ships a full static-file server alongside the Nostr WebSocket). So every church relay automatically becomes a module mirror — no extra moving parts.

For the pro Catalogue enrichment is done by scripts/build-catalog.mjs as part of the release pipeline. The script probes each candidate host (calls /catalog.json and /blossom/list) and only lists hosts that actually respond — so a relay still on the older gateway code doesn't get falsely advertised. Integrity verification has always existed in engine.js installModule() (verifyIntegrity, KNOWN_HASHES); the change extends it to every entry, not just the bundled defaults.

Phase 2 — Blossom (hash-addressed storage)

Blossom is a small HTTP convention from the Nostr ecosystem: ask a server for /, get the file whose contents match that hash, or a 404. The hash is the address. Any server holding that hash serves it; the client doesn't care which.

# Request:
GET https://relay.somechurch.org/blossom/8a2a130d8e0f...

# Response:
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 4012877
ETag: "8a2a130d8e0f..."
Cache-Control: public, max-age=31536000, immutable
<the actual JSON bytes>

Why this is better than just listing more HTTP mirrors:

  • Any church that opts in becomes a mirror, automatically. If a church's relay holds the bytes for 8a2a130d..., it serves them. No central registry of mirrors to keep updated.
  • It looks like a plain HTTPS image fetch. A censor's deep-packet inspector sees a normal HTTPS GET to a randomly-named path — there's no protocol signature to match on.
  • The hash is the verification. Even if a hostile mirror serves wrong bytes, the client rejects them at the moment of receipt.
For the pro Implementation is ~50 lines in scripts/gateway.mjs: routes for GET /blossom/list (monitoring; returns sha→size map), GET /blossom/<sha256> (streams the matching file with Cache-Control: immutable, ETag = the sha), and GET /blossom/rebuild (ops endpoint to refresh the index without restarting). The in-memory index is built at startup by hashing every file in modules/ — cheap for our size (under 50 MB total).

Phase 2 also — the catalogue itself rides on Nostr

The bytes being hash-addressed is half the story. The other half: the list of what's available shouldn't depend on one HTTP host either. So at release time, the build pipeline publishes the catalogue as a signed Nostr event from a dedicated content key. The member app subscribes to that event across the same relays it uses for chat. Block the static HTTP host entirely, and the catalogue still arrives — because it's just another signed message on Nostr.

{
  "kind": 30078,
  "tags": [
    ["d", "catalog:trinityone"],
    ["t", "trinityone"],
    ["version", "2"]
  ],
  "content": "<the full catalog.json as a string>",
  "pubkey": "e328e198...",
  "sig": "..."
}

The pubkey of the catalogue signer is hardcoded in engine.js, so a member's app verifies the signature before trusting the content. Even a malicious relay can't substitute the catalogue (signature would fail) nor swap module bytes (the SHA-256 in the catalogue is the authority, verified after download).

For the pro NIP-78 addressable event; one event per {author, d-tag} pair, latest created_at wins. Signed by the catalogue secret in relay/catalog-key.json (gitignored, on the release host only). Verified client-side via window.Fellowship.verifyEvent (schnorr/secp256k1, exposed from nostr-tools for this purpose). The relay has a write-policy carve-out (scripts/gateway.mjs accept()) so only events from the catalogue pubkey can write to d=catalog:trinityone — preventing a junior steward or anyone else from accidentally publishing a fake catalogue.

Layer 3: Getting the app in the first place

Blossom + signed Nostr catalogue handle the "once installed, what keeps working" question. They don't help someone who's never installed the app and can't reach the marketing site.

For that, TrinityOne's design leans on a different property: every church relay also serves the entire app. The same gateway process that hosts the church's chat data also hosts the welcome page, the member APK, the steward console, the install scripts — all of it. So a member in a network where trinityone.church is blocked can install from any TrinityOne church's relay URL — same APK, same signing key, byte-for-byte identical.

We've shipped a public page explaining this to anyone who needs it: Install TrinityOne from any church. Worth reading; it's the simplest answer to "how does a new install happen in a blocked region."

The honest limit: iPhone PWA install requires Safari to reach some URL at least once. If every TrinityOne hostname is blocked simultaneously, iPhone users need a VPN for the first install. This is true of every PWA-only app on iOS — Apple's design, not ours — and there's no software answer at the app level.

What this defends against — and what it doesn't

ScenarioWhat happens
A church's primary relay goes down for 2 hoursChat keeps working on the other relays the client knows. Phone cache means recent messages are local anyway.
trinityone.church blocked at the DNS layer in a regionCatalogue arrives via Nostr (any reachable relay). Module bytes arrive via Blossom from any reachable mirror. Install works from any church's relay URL (manual sharing).
A hostile mirror serves wrong bytes for a BibleClient SHA-256 check fails before parsing; next mirror tried; nothing tampered ever loads.
A hostile relay publishes a fake catalogueSignature against the hardcoded catalogue pubkey fails; client falls back to HTTP catalogue or the previously-cached one.
Cloudflare disables our Pages projectStatic app + APK still served from every church's relay. The marketing page goes dark; members already installed are unaffected.
Every TrinityOne hostname blocked + every relay blocked simultaneouslyMembers already installed keep the app working on their phone's cache; chat doesn't sync until a relay is reachable. Re-seeding the network requires a single tunneled relay back online.
App Store removaln/a — TrinityOne doesn't ship through stores. The threat doesn't exist.
Total internet shutdown in a regionNo software answer. LAN-only relays (planned, see SPINE) would let a church gathering keep working internally; nothing reaches outside.
State coerces a relay operator into logging downloadsNot defended against by Blossom — the operator sees the IPs. Mitigation is operational (church runs its own relay so logs stay local) or layered (Tor — not in scope).

How you'd actually test the Blossom layer

This is the genuinely hard part. The whole point of Blossom is invisible-in-the-happy-path — so functional testing teaches you "the code parses." To learn whether the design works, you have to deliberately break the normal path and see what fills in. Three tiers, by realism cost:

  1. Unit-level (dev box, today). Tamper a byte in a Blossom response → client sha-mismatch rejects it. Bad catalogue signature → rejected. One source down → next tried. These are done in the headless smoke test that ships with the branch.
  2. Multi-host (needs deploy). Update master-01 so it also has Blossom routes. Now there are two real Blossom mirrors. Stop one → install still works on a device. This tests Phase 1's redundancy.
  3. Censorship simulation (the only one that proves Phase 2). Make the dev box unreachable from a test device while leaving another relay reachable. Watch the catalogue arrive over Nostr, watch bytes arrive from the surviving mirror. Practical ways to do it: /etc/hosts on the device pointing the blocked hostname at 0.0.0.0, DevTools Network request-blocking in Chrome, iptables drops on a Linux test box, or a VPS in a region your relays don't reach.

The honest constraint: Tier 2 and Tier 3 both need at least two working Blossom mirrors. Until master-01 has the updated gateway code, the dev box is the only one — so killing it teaches you "things break" rather than "Blossom catches the fall." The fix is one git pull + restart on master-01 once the branch is ready to ship.

Glossary — terms you'll see in this doc and the code

Relay
A small server that stores and re-broadcasts Nostr events. Stateless about users; just plumbing. More.
Nostr
"Notes and Other Stuff Transmitted by Relays" — an open protocol for signed messages. Each user has a keypair; each message is a JSON event signed by their secret key.
kind:30078 / NIP-78
A type of Nostr event used for "addressable" app data — a tuple of (author, d-tag) defines a slot, and the latest event in that slot is the current value. TrinityOne uses this extensively for church-signed docs (groups, members, funds, the module catalogue, etc.).
Module
A downloadable file: a Bible translation, a dictionary, a commentary, a devotional. Cached on the device after first download.
Catalogue (catalog.json)
The list of modules a member can install. Originally a static file on the gateway; in the Blossom design, also published as a signed Nostr event so it can ride over any relay.
Mirror
An additional host that serves a copy of a file. In Phase 1, listed by URL per catalogue entry. In Phase 2, listed as a Blossom server that holds the file's SHA-256.
SHA-256
A cryptographic hash function. We use it to identify file content and to verify a download wasn't tampered with. Two different files can't have the same SHA-256 (modulo astronomical odds).
Blossom
A small HTTP convention from the Nostr community for hosting binary files addressed by their SHA-256 hash. GET / returns the bytes; any server holding them can serve.
Deploy key
An SSH key tied to a specific repository (not a user account). Holding the private key lets you push to that repo. TrinityOne uses one on the dev box for pushing to github.com/TrinityOneAdmin/TrinityOne.
Tailscale Funnel
A free service that exposes a local server to the public internet via a stable *.tailbeaac0.ts.net URL, with HTTPS. The pilot's stopgap for "how do we make these relays reachable from anywhere" before churches set up their own domains.
Capacitor
The framework that wraps TrinityOne's web app as a native Android APK / iOS PWA. Lets one codebase ship to all three platforms.
